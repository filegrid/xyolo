# 训练目标评估指标体系设计

## 1. 当前状态

目前**还没有一套完整、可执行的评估指标体系文档**。

现有 `training-page.md` 里已经出现了三个相关概念，但都还停留在设计占位层：

1. 可以给节点挂多个验证集；
2. 结果页要展示多验证集指标；
3. 目标对象里有 `best_node_id` 字段，暗示未来需要“当前最优模型”。

但目前**还没有明确规定**：

- 每种任务到底看哪些指标；
- 多个验证集之间如何加权；
- 什么叫“通过”；
- 什么叫“退化”；
- `best_node_id` 应该按什么规则计算。

所以答案是：

- **现在有评估相关概念，但还没有完整指标体系；**
- **现在有最优模型字段占位，但还没有明确的最优判定规则。**

本文档用于单独定义这套体系。

## 2. 设计目标

这套评估体系要解决四个问题：

1. 每个训练节点训练完成后，如何得到一组可解释的评估结果；
2. 多个验证集同时存在时，如何判断本次训练整体是否更优；
3. 如何自动给出“当前最优模型”；
4. 如何避免某个专项集提升、但老能力明显退化时仍被误判为最优。

## 3. 适用范围

适用于训练目标树中的每一个训练节点。

支持三类任务：

1. `detect`
2. `segment`
3. `pose`

不同任务的主指标不同，但整体评估流程一致；不同工具链的原始结果格式也可以不同，但都要先归一化后再进入本体系。

## 4. 基本原则

### 4.1 不是只看单个 mAP

不能只看一个总分，必须至少同时考虑：

- 主验证集表现；
- 回归验证集是否退化；
- 专项验证集是否真正提升；
- 是否存在明显短板。

### 4.2 最优模型不是“最高分模型”这么简单

一个节点要成为当前最优模型，至少要满足：

1. 关键验证集不过度退化；
2. 主目标分数足够高；
3. 没有明显失败项；
4. 综合分优于其他候选节点。

### 4.3 多验证集必须有角色区分

不同验证集不能一视同仁，需要区分角色：

1. **主验证集**：最核心的业务指标；
2. **回归验证集**：确保旧能力不明显退化；
3. **专项验证集**：验证某次分支新增价值；
4. **观察验证集**：只展示，不参与最优判定。

## 5. 指标层级

评估分成三层：

1. **原始指标**
2. **验证集级评分**
3. **节点级综合评分**

### 5.1 原始指标

每次评估都先保留模型直接产出的原始指标。

这些原始指标**不要求所有工具链字段名完全一致**。公共层只要求工具链适配层最终产出统一的 `normalized_metrics`。

#### detect

- `map50_95`
- `map50`
- `precision`
- `recall`
- `per_class_ap`

#### segment

- `box_map50_95`
- `mask_map50_95`
- `precision`
- `recall`
- `per_class_ap`

#### pose

- `pose_map50_95`
- `pose_map50`
- `box_map50_95`
- `precision`
- `recall`

### 5.2 任务主指标

每类任务都必须指定一个**主指标**，用于综合排序时的核心比较。

推荐规则：

| 任务类型 | 主指标 | 次指标 |
|---|---|---|
| detect | `map50_95` | `precision` / `recall` / `map50` |
| segment | `mask_map50_95` | `box_map50_95` / `precision` / `recall` |
| pose | `pose_map50_95` | `box_map50_95` / `precision` / `recall` |

说明：

- `detect` 以检测总体精度为主；
- `segment` 应优先看分割质量，而不是只看框；
- `pose` 应优先看关键点质量。

### 5.3 验证集级评分

每个验证集都计算一个 `dataset_score`，统一归一到 `0 ~ 1`。

推荐默认公式：

```text
dataset_score =
  0.70 * main_metric +
  0.15 * precision +
  0.15 * recall
```

其中：

- `main_metric` 由任务类型决定；
- 所有输入都使用 `0 ~ 1` 小数值；
- 如果某任务没有完全对应的 `precision/recall`，则使用该任务已有的等价统计项。

这个分数的意义不是替代原始指标，而是为了支持多验证集聚合和节点排序。

在工具链架构下，推荐流程是：

1. 工具链输出 `provider_result.raw_metrics`；
2. adapter 转成 `normalized_metrics`；
3. 评估策略基于 `normalized_metrics` 计算 `dataset_score`。

## 6. 验证集角色与权重

每个验证集除了路径，还必须带角色和权重。

建议结构：

```json
{
  "name": "night-val",
  "role": "scenario",
  "weight": 0.20,
  "is_required": true
}
```

角色定义：

| role | 含义 | 是否默认参与最优模型计算 |
|---|---|---|
| primary | 主验证集 | 是 |
| regression | 回归验证集 | 是 |
| scenario | 专项验证集 | 是 |
| observe | 观察验证集 | 否 |

权重建议：

| 角色 | 推荐权重范围 |
|---|---:|
| primary | 0.40 ~ 0.60 |
| regression | 0.20 ~ 0.40 |
| scenario | 0.10 ~ 0.30 |
| observe | 0 |

约束：

1. 至少有一个 `primary`；
2. 参与排序的验证集权重总和必须等于 `1`；
3. `observe` 集不参与总分；
4. 一个节点可以比目标默认值额外增加 `scenario` 集，但不能移除必需的 `primary/regression` 集。

## 7. 节点级综合评分

### 7.1 基础综合分

节点综合分 `node_score` 由参与排序的验证集加权得到：

```text
node_score = Σ(dataset_score_i * validation_weight_i) - penalties
```

其中：

- `dataset_score_i` 是单验证集分数；
- `validation_weight_i` 是该验证集权重；
- `penalties` 是退化、失败、风险等惩罚项。

### 7.2 硬性门槛

在进入排序前，先做硬性过滤。

只要触发以下任一条件，该节点就**不能成为 best model**：

1. 主验证集评估失败或缺失；
2. 任一必需验证集评估失败；
3. 主指标低于目标配置的最低门槛；
4. 回归验证集相对父节点退化超过阈值；
5. 产出模型文件不存在或不可用。

推荐默认阈值：

| 项目 | 默认值 |
|---|---:|
| 主验证集主指标最低门槛 | 0.30 |
| 回归集允许退化上限 | -0.02 |
| 主验证集允许退化上限 | -0.01 |

这里的 `-0.02` 表示相对父节点主指标下降超过 2 个百分点。

### 7.3 惩罚项

未触发硬性失败，但存在风险时，仍应扣分。

推荐惩罚：

| 场景 | 惩罚 |
|---|---:|
| 某个 regression 集略微下降但未超硬阈值 | 0.01 ~ 0.03 |
| 某个 required scenario 集提升不明显 | 0.01 |
| 某个验证集明显偏科，precision/recall 失衡 | 0.01 ~ 0.02 |
| 节点训练结果不完整，缺少部分统计 | 0.05 |

## 8. 与父节点的比较维度

每个节点评估时，不只看绝对值，也必须看相对父节点变化。

至少输出以下差异：

1. `delta_main_metric`
2. `delta_precision`
3. `delta_recall`
4. `delta_node_score`

推荐展示结论：

| 条件 | 结论 |
|---|---|
| 主验证集提升，回归集稳定 | 通过 |
| 主验证集提升，但回归集略退化 | 有提升但旧集退化 |
| 专项集提升很弱 | 新场景提升不明显 |
| 新类别集提升明显，旧集稳定 | 新类别有效，旧类别稳定 |
| 主验证集或回归集明显下降 | 整体回归失败 |

## 9. 最优模型判定规则

### 9.1 目标

每个训练目标都应自动维护：

```json
{
  "best_node_id": "node-002"
}
```

这个字段表示该目标下**当前推荐默认使用的模型**。

### 9.2 候选集合

参与最优模型评选的节点必须满足：

1. `status = completed`
2. 有可用模型文件
3. 全部必需验证集评估完成
4. 未触发硬性门槛失败

### 9.3 排序规则

候选节点按以下优先级排序：

1. `node_score` 高者优先；
2. 若 `node_score` 相同，则主验证集主指标高者优先；
3. 若仍相同，则 regression 集最差退化更小者优先；
4. 若仍相同，则专项验证集覆盖更多者优先；
5. 若仍相同，则更新时间更晚者优先。

### 9.4 为什么不能只按单个 primary 排序

如果只按主验证集最高分排序，会出现两个问题：

1. 新场景变好，但老场景明显退化；
2. 某条分支对专项集过拟合，却不适合作为默认模型。

所以 `best_node_id` 必须根据综合规则产生，而不是简单取某一个 mAP 最高的节点。

## 10. 最优模型的多视角输出

除了全局 `best_node_id`，建议再支持两个扩展字段：

```json
{
  "best_node_id": "node-002",
  "best_by_validation": {
    "base-val": "node-001",
    "night-val": "node-002"
  },
  "best_by_metric": {
    "map50_95": "node-002"
  }
}
```

含义：

1. `best_node_id`：全局推荐模型；
2. `best_by_validation`：某个验证集上的局部最佳；
3. `best_by_metric`：某个指标维度上的最佳。

这样可以同时回答：

- “当前默认应该用哪个模型？”
- “夜间场景哪个模型最好？”
- “哪个模型总 mAP 最高？”

## 12. 页面展示建议

### 12.1 节点卡片

至少展示：

- 主验证集主指标；
- 综合分 `node_score`；
- 相对父节点变化；
- 最差验证集；
- 评估结论；
- 是否为当前 `best model`。

### 12.2 目标详情

目标详情页应单独展示：

1. 当前推荐模型；
2. 推荐原因；
3. 候选节点排行榜；
4. 各验证集局部最佳；
5. 风险提示。

### 12.3 最优模型标识

在训练树中，`best_node_id` 对应节点应有显式标记，例如：

- `Best`
- `Recommended`
- `当前默认模型`

避免用户只能自己看表格猜。

## 13. 推荐实施顺序

### P0

- 定义验证集角色；
- 落盘单验证集原始指标；
- 定义任务主指标；
- 生成 `dataset_score`；
- 生成 `node_score`；
- 自动计算 `best_node_id`。

### P1

- 增加相对父节点差异分析；
- 增加 gate 与 penalty；
- 页面展示推荐原因和风险提示；
- 支持 `best_by_validation`。

### P2

- 支持不同任务类型自定义评分公式；
- 支持类别级评分；
- 支持更细粒度的业务权重模板。

## 14. 最终结论

这套体系的核心不是“再多记几个指标”，而是把评估拆成三层：

1. **原始指标**：保留真实训练输出；
2. **验证集级评分**：解决多验证集可比较问题；
3. **节点级综合评分**：解决最优模型自动推荐问题。

同时，工具链层和评估层分工明确：

1. 工具链层负责跑训练/评估并产出原始结果；
2. 归一化层负责把 YOLO、未来 Torch 等结果映射到统一字段；
3. 评估策略层负责排序、告警和最优模型判定。

落地后可以明确回答两个关键问题：

1. **目前有评估指标体系吗？**
   - 现在还没有完整体系，本文档定义了它。
2. **有根据指标显示最优模型吗？**
   - 现在只有 `best_node_id` 概念占位；按本文档落地后，可以自动基于指标显示当前最优模型。
